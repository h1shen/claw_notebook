# Claude Code 使用指南

## 法则一：先思考，再输入 - 计划模式是你最强大的武器

- **计划模式**（Shift+Tab 连按两次）的输出远胜于直接输入提示词
- 真正至关重要的第一步是思考和计划，而非直接开始编码

### 两条建议
1. **开始学习**：即使每次只学一点，也必须开始积累规划能力
2. **与AI深度交流**：和Claude进行双向对话，而不是单向输入

---

## 法则二：CLAUDE.md 不是文档，而是AI的大脑

CLAUDE.md 是一个极其重要但常被误用的配置文件。Claude 在每次启动会话时都会首先读取它。

### 编写 CLAUDE.md 的四个关键法则

1. **保持简短**
   - Claude 一次只能可靠地遵循约 150-200 条指令
   - 系统提示本身已占用约 50 条指令
   - 如果写得太长，Claude 会随机忽略某些内容

2. **专注于项目特性**
   - 不要描述通用概念（如"组件文件夹"）
   - 重点说明项目中的"奇怪"之处（特有的bash命令、工作流程）

3. **解释"为什么"，而不仅是"做什么"**
   - 例：不只说"使用TypeScript严格模式"
   - 而是说"使用TypeScript严格模式，因为我们曾因隐式any类型导致过生产环境的bug"
   - 这样Claude能更好地理解意图并做出更优判断

4. **持续更新**
   - CLAUDE.md 应该是"活文档"
   - 每当发现自己第二次纠正Claude同一问题时，就应该写入CLAUDE.md
   - 优秀的CLAUDE.md 读起来像是给失忆的自己留下的核心笔记，而不是入职文档

---

## 法则三：200k 上下文是甜蜜的陷阱，别掉进去

### 关键事实
- 模型性能下降远在上下文窗口被完全填满前就开始了
- 通常在使用率达到 20-40% 时就会出现明显衰减
- 即使压缩上下文（/compact），也可能因为之前的退化而输出糟糕

### 有效管理上下文的实用策略

1. **划分对话范围**
   - 每个功能或任务使用独立的对话
   - 不要在同一对话里既构建认证系统又重构数据库层

2. **使用外部记忆**
   - 让Claude将计划和进度写入 SCRATCHPAD.md 等外部文件
   - 第二天回来时，Claude 可以读取文件，从上次中断处继续

3. **"复制-粘贴"重置法**
   - 复制对话中的关键信息
   - 运行 /compact 和 /clear 清空上下文
   - 只把最重要的信息粘贴回来
   - 清爽的上下文远胜于退化的上下文

4. **果断清空**
   - 如果对话已经偏离轨道，直接用 /clear 重新开始
   - 这通常比试图纠正混乱对话更有效

### 心智模型
**Claude 是无状态的。** 除了你明确给它的东西，每个对话都是从零开始。

## 法则四：架构决定一切，规划无可替代

### 核心原则
- 架构至关重要，尤其是在软件工程中
- 如果不先思考结构，AI生成的代码会有巨大的"自由发挥"空间
- **你不能跳过规划**

### 具体 vs 模糊请求对比

**模糊的请求：**
```
给我建一个认证系统
```

**经过规划的、具体的请求：**
```
使用现有的User模型构建电子邮件/密码认证功能，
将session存储在Redis中并设置24小时过期，
并添加中间件保护/api/protected下的所有路由。
```

**差异：**
- 前者给了AI过多自由，结果可能混乱
- 后者给了清晰的蓝图，结果会精准得多
- 花5分钟架构规划，可以省下后续数小时的调试时间

---

## 法则五：停止抱怨模型，糟糕的输出源于你糟糕的输入

### 残酷的现实
- 如果用 Opus 4.5 还得不到好结果，**问题出在你身上，而不是AI**
- 你的输入和提示方式决定了输出质量

### 三个提升输入质量的方法

1. **具体说明你想要什么**
   - 指令越清晰、越具体，结果就越好

2. **告诉它不要做什么**
   - Claude 4.5 有过度设计的倾向
   - 明确说："保持简单，不要添加我没要求的抽象，如果可能的话，只用一个文件"

3. **提供"为什么"的背景**
   - 告诉它"这个功能需要在每个请求上运行，所以性能至关重要"
   - 或"这只是一个原型，用完就扔"
   - 这些约束会彻底改变模型解决问题的思路

### 专家级工作流
- 用 **Opus** 进行规划和架构设计
- 用 **Sonnet** 进行具体实现
- Opus 更擅长复杂推理，Sonnet 更快、更便宜，适合执行明确任务

### 记住这个真理
> **如果你的输出很糟糕，那是因为你的输入很糟糕。没有捷径可走。**

## 法则六：勇于实验，配置决定你的上限

### Claude 的功能生态系统

1. **MCP (Model Context Protocol)**
   - 让Claude连接到外部服务（Slack、GitHub、数据库）
   - 如果总是在复制粘贴信息，可能有MCP服务器能帮你自动化

2. **Hooks**
   - 让代码在Claude修改前后自动运行
   - 想让Prettier格式化文件？用Hook
   - 想在编辑后进行类型检查？用Hook
   - 能立即捕获问题

3. **自定义斜杠命令**
   - 把重复使用的提示词打包成命令
   - 在 .claude/commands 文件夹里创建markdown文件
   - 然后用 /commandname 来运行

### 保持好奇心
- 这些模型每周都在进步
- 一个月前行不通的功能，现在可能已经可以了
- 不断重新测试已有功能

---

## 法则七：当你被卡住时，停止强推，改变方法

### 识别死循环
- Claude陷入循环：尝试、失败、再尝试、再失败
- 人的本能是继续解释、提供更多指令
- **但更好的做法是彻底改变方法**

### 四个改变方法的策略

1. **清空对话**
   - 累积的上下文可能正在迷惑它
   - 一个全新的开始可以解决问题

2. **简化任务**
   - 如果复杂任务让Claude举步维艰，分解成更小的部分
   - 如果Claude处理复杂任务很吃力，通常意味着初始计划不充分

3. **展示而非告知**
   - 如果Claude一直无法理解，亲手写一个最小化的正确示例
   - 告诉它："看，最终输出应该像这样。现在把这个模式应用到其他部分。"

4. **重构问题**
   - 换一个角度来描述问题
   - 原始表述方式可能不符合Claude的"思维模型"
   - 如果重复解释了三遍，是时候改变策略了

---

## 法则八：超越聊天模式，构建自动化系统

### 真正的价值所在
- 真正从Claude中获得巨大价值的人，**不仅仅把它当作交互式工具**
- 他们正在构建以Claude为核心组件的自动化系统

### 无头模式 (Headless Mode)
- 通过 -p 标志在无头模式下运行Claude
- 可以编写脚本，将输出通过管道传递给其他工具
- 与bash命令链接，集成到自动化工作流中

### 企业用例
- 自动化代码审查（PR review）
- 支持工单响应
- 日志记录和文档更新
- 所有这些都是可记录、可审计的，并随时间改进

### 复合改进的飞轮效应
1. Claude犯了一个错误
2. 你审查日志
3. 改进CLAUDE.md或相关工具
4. 下一次Claude就会做得更好
5. 这种改进是复合式的

### 注意
- **如果只在交互模式下使用Claude，你正在错失它真正的价值**

---

## 结语：你的AI，你的责任

### 思维模式的转变

真正掌握像Claude这样的AI开发工具，关键在于思维模式的转变：

| | |
|---|---|
| **从** | 简单的指令发出者 |
| **转变为** | 严谨的规划者、配置者和系统构建者 |

### 核心认知
> **你不是在和它聊天，你是在编程它。**

### 最后的思考题
> 如果你不把Claude当作聊天机器人，而是看作一个可编程的团队成员，你最先会自动化工作流程的哪个部分？
