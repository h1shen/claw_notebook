# 架构规划：AI 辅助编程中的规划思维

🟡 个人实践

## 核心原则

架构至关重要，尤其是在软件工程中。如果不先思考结构，AI 生成的代码会有巨大的"自由发挥"空间。**你不能跳过规划。**

## 具体 vs 模糊请求对比

**模糊的请求：**
```
给我建一个认证系统
```

**经过规划的、具体的请求：**
```
使用现有的 User 模型构建电子邮件/密码认证功能，
将 session 存储在 Redis 中并设置 24 小时过期，
并添加中间件保护 /api/protected 下的所有路由。
```

**差异：**
- 前者给了 AI 过多自由，结果可能混乱
- 后者给了清晰的蓝图，结果会精准得多
- 花 5 分钟架构规划，可以省下后续数小时的调试时间

## 规划先于执行

在让 Claude 写代码之前，先在脑海中（或纸上）回答：

1. 用什么数据结构存储状态？
2. 各模块之间如何通信？
3. 哪些是外部依赖，哪些自己实现？
4. 错误处理策略是什么？

清晰的架构意图会让 Claude 的输出质量大幅提升。
